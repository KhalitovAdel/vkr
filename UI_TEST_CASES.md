# UI Test Cases (для ручной проверки)

## Как открывать URL

В React Router пути **без префикса** `/#/` (React Router v6): полный адрес в браузере обычно выглядит так:

- **Только бэкенд (jar / `./mvnw`)**: `http://localhost:8080/dashboard`, `http://localhost:8080/trip/suggestion` и т.д.
- **Dev-режим (webpack на 9000 + прокси на API)**: тот же путь после хоста, например `http://localhost:9000/trip/suggestion` — см. `README.md` про `./npmw run backend:start` и `./npmw run start`.

Меню сущностей: пункты вроде **Trip Suggestion**, **Vehicle**, **Route** — в боковом меню после входа (см. `src/main/webapp/app/entities/menu.tsx`).

## Глоссарий (важно для кейсов)

| Термин | Что это | Где в UI |
|--------|---------|----------|
| **Route (маршрут)** | Справочная сущность: линия маршрута, тип, параметры. Не привязан к конкретной дате/времени. | Список: `/route`, карточка: `/route/:id`, создание: `/route/new` |
| **Trip (рейс)** | Конкретный выезд: дата, время прибытия/отправления, ТС, водитель, статус, связь с waybill и маршрутом. | Список: `/trip`, создание: `/trip/new` |
| **`routeId` в подборе ТС** | Числовой **ID маршрута** из таблицы маршрутов (`/route`), **не** ID рейса. Нужен алгоритму, чтобы понять требования к вместимости и логику подбора. | Взять из списка `/route` (колонка ID) или из URL карточки `/route/5` → `routeId = 5` |
| **Waybill (путевой лист)** | Документ с фактическим выездом/возвратом, пробегом и т.д., обычно связан с рейсом. | Список: `/waybill`, правка: `/waybill/:id/edit` |

## Пререквизиты

- Пользователь залогинен под ролью с доступом к сущностям (в проекте — `USER` / `ADMIN`, см. `PrivateRoute` в `routes.tsx`).
- Для кейсов с данными: заранее есть хотя бы один **маршрут** (`/route`), **ТС** (`/vehicle`), при необходимости **водитель** (`/driver`).
- Для проверки пересечений: два рейса или подбор после создания первого рейса.

## Кейсы

### 1. Загрузка главной страницы (Dashboard)

| Поле | Описание |
|------|----------|
| **URL в UI** | `/dashboard` |
| **Зачем** | Проверить, что отчёт по парку (`GET /api/reports/fleet`) отображается в UI без ошибок. |
| **Когда** | Smoke-тест после деплоя/запуска; после массовых изменений в справочнике ТС. |
| **Шаги** | Открыть URL, дождаться окончания загрузки (не только спиннер «Загрузка дашборда…»). |
| **Ожидаемо** | Отображаются «Всего ТС», «Исправных», «В ремонте»; нет 5xx/пустой страницы из-за необработанной ошибки. |

---

### 2. Автоподбор ТС по форме

| Поле | Описание |
|------|----------|
| **URL в UI** | `/trip/suggestion` (в меню: **Trip Suggestion**) |
| **Зачем** | Это **помощник планирования**: до создания рейса получить рекомендацию «какое ТС подойдёт» по маршруту, дате и времени выезда (вместимость, статус, занятость — по логике бэкенда). |
| **Когда** | Перед созданием рейса на `/trip/new`; при смене парка (ремонт, новые машины); регресс после изменений в `TripService` / `POST /api/trips/suggest-vehicle`. |
| **Откуда `routeId`** | **Не** ID рейса. Открой `/route`, выбери нужный маршрут — его **ID** в таблице или в URL `/route/{id}`. |
| **Шаги** | Ввести `routeId`, `tripDate`, `departureTime` → нажать кнопку подбора (в UI обычно «Подобрать» / аналог). |
| **Ожидаемо** | Показано предложенное ТС **или** понятное сообщение, что подходящих вариантов нет (без необработанного исключения в UI). |

### 3. Исключение занятых ТС (пересечение времени)

| Поле | Описание |
|------|----------|
| **URL в UI** | Подготовка: `/trip/new` → проверка: `/trip/suggestion` |
| **Зачем** | Убедиться, что машина, уже занятая рейсом на пересекающийся интервал, **не попадает** в подбор (или не единственный вариант — по ТЗ). |
| **Когда** | После реализации `findOccupiedVehicleIds` / логики пересечений в репозитории/сервисе. |
| **Шаги** | 1) Создать рейс с известным `vehicle` и временем. 2) На те же (или пересекающиеся) `tripDate`/`departureTime` открыть `/trip/suggestion` с тем же `routeId` (если требуется сценарий). |
| **Ожидаемо** | Занятая машина не предлагается или предлагается другое ТС. |

---

### 4. Исключение занятых водителей (пересечение времени)

| Поле | Описание |
|------|----------|
| **URL в UI** | `/trip/new`, при необходимости `/trip/suggestion` |
| **Зачем** | Проверка антидублирования по **водителю** на пересекающиеся рейсы. |
| **Когда** | После доработки проверок при создании рейса и/или подборе. |
| **Шаги** | Создать рейс с водителем A на интервал T; попытаться создать второй рейс с тем же водителем на пересечение с T (или прогнать сценарий подбора, если UI это поддерживает). |
| **Ожидаемо** | Система не даёт двойной назначение; сообщение об ошибке понятно. |

### 5. Валидация `arrivalTime` и `departureTime`

| Поле | Описание |
|------|----------|
| **URL в UI** | `/trip/new` |
| **Зачем** | Соблюдение бизнес-правила: время прибытия **позже** времени отправления. |
| **Когда** | Проверка валидаций сущности Trip (п. 6.1 ТЗ). |
| **Шаги** | Указать `arrivalTime` **раньше или равно** `departureTime` (некорректная пара). |
| **Ожидаемо** | Ошибка валидации, рейс не сохраняется. |

---

### 6. Запрет создания рейса при `technicalStatus = REPAIR`

| Поле | Описание |
|------|----------|
| **URL в UI** | `/vehicle/:id/edit` → `/trip/new` |
| **Зачем** | ТС в ремонте не должно участвовать в новом рейсе (п. 6.2 ТЗ). |
| **Когда** | После добавления проверки на бэкенде при создании Trip. |
| **Шаги** | У выбранной машины выставить **Technical status = REPAIR**, сохранить. Создать новый рейс, выбрав эту машину. |
| **Ожидаемо** | Создание блокируется, есть явное сообщение. |

### 7. Валидация стажа водителя (меньше 3 лет на «сложном» маршруте)

| Поле | Описание |
|------|----------|
| **URL в UI** | `/driver/new` или `/driver/:id/edit` → `/trip/new` |
| **Зачем** | Правило допуска водителя по стажу для сложных маршрутов (п. 6.3 ТЗ; конкретика «сложный» — по типу/атрибуту Route в коде). |
| **Когда** | После реализации правила в сервисе Trip. |
| **Шаги** | Водитель со стажем меньше 3 лет (поле **Experience**), маршрут, который система считает сложным, — попытка создать рейс. |
| **Ожидаемо** | Отказ с сообщением про стаж/категорию маршрута. |

---

### 8. Предотвращение двойного назначения (ТС уже занято)

| Поле | Описание |
|------|----------|
| **URL в UI** | `/trip/new` (два раза подряд) |
| **Зачем** | Контроль занятости ТС при **создании** второго рейса (п. 2.1 ТЗ), не только в подборе. |
| **Когда** | Регресс после изменений в `TripService` при save/create. |
| **Шаги** | Рейс 1: машина M, дата/время. Рейс 2: та же M, пересекающийся интервал. |
| **Ожидаемо** | Второй рейс не создаётся; ошибка связана с занятостью ТС. |

### 9. Учёт выхода на линию (Waybill — departure)

| Поле | Описание |
|------|----------|
| **URL в UI** | `/waybill/:id/edit` (ID из списка `/waybill` или ссылки с рейса `/trip`) |
| **Зачем** | Фиксация фактического выезда и начального пробега (п. 3.1, 3.4 ТЗ — если в UI вызов идёт через форму; при отдельных кнопках API — проверять и их). |
| **Когда** | После реализации `registerDeparture` и связки с Trip. |
| **Шаги** | Заполнить поля выезда (`actualDeparture`, `mileageStart` и т.д. по форме), сохранить. |
| **Ожидаемо** | Данные сохранены; при наличии логики — статус рейса сдвигается к «в пути» (`ONGOING`). |
